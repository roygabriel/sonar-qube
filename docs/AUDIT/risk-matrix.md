# Risk Matrix

| Risk | Likelihood | Impact | Trigger | Mitigation | Gate | Owner |
|---|---|---|---|---|---|---|
| Phase 1: Core MCP server functionality drift from reference (main.go, pkg/server/handlers.go, config.go) | Medium | High | Divergence in Go handler logic or config parsing vs cruvero/cruvero-mcp-k8s@dev reference | Perform side-by-side binary comparison tests and `go test ./pkg/server/... -race`; enforce strict code diff reviews for key functions; monitor via CI diffs and behavior logs on every commit | phase-audit-1 | MCPArchitect |
| Phase 2: Dedicated testing phase defects in adapted Go components | Medium | Medium | Test suite failures or critical defects in server handlers/config during validation | Execute `make test` with integration tests simulating K8s API calls; require 90%+ coverage on adapted files; monitor with automated coverage reports and failure alerts in CI dashboard | phase-audit-2 | GoSpecialist |
| Phase 3: Helm deployment failures from deployment/helm/ in target cluster | Medium | High | Incompatible values.yaml or templates/deployment.yaml with cluster API versions or resource constraints | Run `helm lint && helm template --values deployment/helm/values.yaml` plus `kubectl apply --dry-run=server`; validate against exact cluster version; monitor with pre-commit hooks, deployment event logs and Prometheus alerts | phase-audit-3 | K8sEngineer |
| Phase 4: Failure handling and rollback deficiencies during K8s rollout | Low | High | Unhandled errors in Go server or Helm hooks without verified rollback to stable state | Test failure paths with chaos-mesh on kind cluster targeting handlers.go logic; use `helm upgrade --atomic` and `kubectl rollout undo`; monitor rollout status via `kubectl rollout status` and pod crash alerts | phase-audit-4 | TestEngineer |
| Phase 5: Incomplete validation of all exit criteria and test results | Medium | Medium | Missing deterministic evidence for ACs or binary test outputs across phases | Automate verification script checking all 5 ACs against reference behavior; run `make test` with JUnit artifacts; monitor via phased CI quality gates requiring green status on all dependency edges | phase-audit-5 | MCPArchitect |
| Dependency coupling: phase-1 → phase-2 | Low | High | Contract drift in Go server outputs (handlers/config) breaking downstream test expectations | Define explicit test contracts for main.go entrypoints; run cross-phase regression suite after phase-1 changes; monitor with automated contract tests and diff reports in pipeline | dependency gate | MCPArchitect |
| Dependency coupling: phase-2 → phase-3 | Low | High | Test-validated configs not aligning with Helm values.yaml or templates/deployment.yaml | Map test-derived resource limits to values.yaml; execute config consistency checker script; monitor by rendering manifests after tests and alerting on parameter mismatches | dependency gate | GoSpecialist |
| Dependency coupling: phase-3 → phase-4 | Low | High | Helm deployment artifacts lacking proper probes/hooks for rollback verification | Add liveness/readiness probes tied to server handlers in templates/deployment.yaml; test rollback in isolated K8s namespace; monitor with deployment metrics and Helm history queries | dependency gate | K8sEngineer |
| Dependency coupling: phase-4 → phase-5 | Low | High | Rollback results not aggregated into final validation evidence | Aggregate all phase logs into single validation report script; enforce traceability from phase-4 tests to AC5; monitor with end-to-end dashboard tracking all 4 dependency edges at 0.1-0.2 risk level | dependency gate | TestEngineer |
